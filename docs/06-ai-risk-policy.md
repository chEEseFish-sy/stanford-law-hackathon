# AI 风险政策

## 产品定位

VeriCap 是面向法律专业人员的 audit copilot。它辅助证据抽取、working cap table 重建、不一致检测和复核优先级排序，但不提供最终法律意见。

## 允许的 AI 输出

- 文件类型建议。
- 事实字段抽取。
- 带来源的证据片段。
- 股权事件候选项。
- working cap table 草稿行。
- 置信度或不确定性提示。
- 冲突或缺失证据建议。
- 对某个结果如何从证据得出的解释。
- 明确标记为草稿的复核笔记。

## 禁止的 AI 输出

- 最终法律意见。
- 交易有效或无效的确定性结论。
- 建议继续交割或停止交割的确定性意见。
- 在未经律师复核的情况下声称 cap table 法律正确。
- 在未经律师复核的情况下认定某方违约。
- 声称已经发现所有相关法律风险。

## 人工复核要求

- 每个关键 cap table 结果在外部依赖前都需要律师或法律专业人员复核。
- 低置信度字段必须标记为需要复核。
- 证据冲突必须升级给人工，而不是自动解决。
- AI 解释必须引用来源证据并说明不确定性。
- 人工决策应与 AI 输出分开记录。

## UI 要求

界面应当：

- 将生成内容标记为工作草稿或复核辅助。
- 对关键值展示来源文件和来源位置。
- 展示不确定性和低置信度状态。
- 区分已确认、已驳回和待复核事项。
- 避免使用“法律正确”“已批准”“可以交割”“最终意见”等措辞。

## Prompt 要求

系统 prompt 和工作流 prompt 应当：

- 要求基于证据抽取，而不是输出法律结论。
- 在证据不足时明确不确定性。
- 对关键字段要求来源引用。
- 禁止最终法律意见。
- 优先输出可审计的结构化结果。

## 需要防范的失败模式

- 编造股份数、投资人、价格或日期。
- 对模糊条款过度自信。
- 把草稿文件当作最终签署文件。
- 忽略文件版本关系。
- 混淆名称相似的法律主体。
- 在没有证据的情况下推断缺失审批。
- 把生成的 cap table 呈现为权威结果。

## Beta 可接受标准

一个结果只有在法律专业人员能理解其来源、生成原因和剩余不确定性时，才适合进入 Beta。
