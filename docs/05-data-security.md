# Beta 数据安全政策

## 目的

本文定义 VeriCap 在被国内律所业内人员试用、或处理真实/脱敏法律文件前，最低限度需要满足的数据处理要求。

## 数据类型

- 上传的法律文件。
- 抽取出的文本和结构化证据。
- working cap table 输出。
- 冲突和风险记录。
- 用户复核决定。
- 操作日志。
- 反馈记录。
- API key 和模型配置。

## 核心原则

- 只使用完成内测验证所必需的数据。
- 首轮测试优先使用合成或脱敏文件。
- 未经明确书面同意，不得将上传文件用于模型训练。
- case 数据应隔离。
- 接收敏感文件前，必须说明保留和删除规则。
- 只允许授权团队成员访问。
- 对关键操作尽量记录日志。

## Beta 数据规则

- 默认使用样例文件、合成文件或脱敏文件。
- 使用真实客户文件前，必须获得内测参与者明确授权。
- 团队必须说明文件存储位置、访问人员和删除时间。
- API key 不得提交到仓库。
- `.env` 文件必须保持本地，并被 git 忽略。
- 中间生成物可能包含敏感文本，应按源文件同等级处理。

## 模型服务商处理要求

在把 Beta 材料发送给任何外部 LLM 服务商之前，必须记录：

- 服务商名称。
- 使用的 API 或产品。
- 上传内容是否会被保留。
- 上传内容是否会被用于训练。
- 如有保留，保留期限是多久。
- 是否存在境内外数据传输或区域问题。
- 内测参与者是否同意这种处理方式。

## 首轮 Beta 访问控制

最低可接受控制：

- 使用受控部署或本地演示环境。
- 只允许明确授权的团队成员访问。
- 不开放公开上传真实敏感文件的入口。
- 不同内测参与者使用不同工作区或存储位置。
- 上传、处理、删除和导出操作应尽量记录。

## 数据保留与删除

对每个内测参与者，应明确：

- 演示结束后是否保留文件。
- 上传文件和处理结果保留多久。
- 如何申请删除。
- 谁负责确认删除。
- 是否可以保留脱敏后的汇总反馈。

## 仓库卫生

- 不要提交用户上传的敏感文件。
- 不要提交生产 API key。
- 不要提交包含客户数据的数据库。
- 样例 fixtures 必须清楚标记为样例、合成或已授权。
- 如果 `storage/` 由真实文件生成，应视为敏感数据。

## 扩大内测前的开放问题

- Beta 是本地运行、私有云运行，还是托管服务器运行？
- 上传律所或客户文件时，谁是数据控制方？
- 产品面向境内部署、境外部署，还是两者都支持？
- 目标律所能接受哪些模型服务商？
- 团队实际能承诺什么删除 SLA？
- 真实文件测试前是否需要用户账号和权限系统？
