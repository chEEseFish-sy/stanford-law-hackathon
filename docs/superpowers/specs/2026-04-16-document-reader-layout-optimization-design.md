# 文档阅读器布局优化设计

## 1. 背景

当前工作台是典型的三栏分析式布局，包含：

- 左侧 `AI Chat`
- 中间混合区域：上传预览、对比卡片、摘要块、cap table 片段
- 右侧上下堆叠的 `History` 与 `Files`
- 左右和上下可拖拽调节尺寸的分隔条

这套结构会把注意力分散到多个模块上，压缩真正的文档文本显示空间。用户希望把页面核心从“分析仪表盘”切换成“文档优先阅读器”。

## 2. 设计目标

将页面改造成以文档阅读为中心的固定布局，让中间区域几乎完全服务于阅读，左右侧栏退居辅助位置。

成功标准：

- 删除右侧 `History` 模块
- 删除全部可拖拽调宽组件，改为固定布局
- 保留 `AI Chat` 与 `Files`，但改成可折叠侧栏
- 左右折叠按钮要在视觉上对称、成对出现
- 中间区域只保留文档阅读能力
- 文档导航从上下滚动改为左右切页
- 文字显示区域尽可能大，因为阅读是核心功能
- 进入页面时默认不自动打开文档
- 没有选中文档时，在阅读器内显示明确的引导型空状态

## 3. 已确认决策

以下方向已经和用户确认：

- 整体方案：`A. 对称折叠轨道`
- 默认侧栏状态：`AI Chat` 展开，`Files` 展开
- 阅读模式：真实分页，使用左右翻页
- 默认阅读内容：进入时先不显示具体文档
- 空状态样式：`A2. 引导操作空状态`

## 4. 信息架构

### 4.1 最终页面结构

页面仍然是单路由工作台，但内容结构调整为：

1. 左侧固定面板：`AI Chat`
2. 中间自适应面板：`Document Reader`
3. 右侧固定面板：`Files`

从主布局中移除：

- `History` 模块
- 中间的 `Previous / Current` 对比双栏视图
- 摘要区和 cap table 区
- 全部横向、纵向拖拽分隔条

### 4.2 各区域职责

`AI Chat`

- 辅助工具，不再是主视觉
- 用于围绕当前文档发问
- 可折叠，为阅读区让出空间

`Document Reader`

- 页面主任务区
- 永远占据最大宽度
- 只显示两种状态：引导空状态或单页文档阅读

`Files`

- 右侧唯一模块
- 承担文档选择与上传入口
- 可独立折叠

## 5. 布局行为

### 5.1 固定宽度侧栏

左右两侧从“用户可调宽”改为“固定宽度”面板。

行为要求：

- 布局内不再保留任何拖拽手柄
- `AI Chat` 使用固定展开宽度
- `Files` 使用固定展开宽度
- 中间阅读器自动填满剩余空间
- 任意一侧折叠后，中间阅读器立即扩张占用释放出的宽度

这样可以减少视觉噪音，让工作台更加稳定。

### 5.2 对称折叠轨道

左右侧栏都支持折叠/展开，交互形式采用镜像式轨道按钮。

要求：

- 左右折叠按钮必须有“成对”的感觉
- 折叠后，面板收成窄轨道
- 轨道上仍保留图标和紧凑标签，保证可发现性
- 左侧轨道向左收起
- 右侧轨道向右收起
- 展开后恢复到原固定宽度

视觉指导：

- 使用镜像的几何形态、内边距和图标位置
- 动效要短、轻、克制
- 折叠过程中，中间阅读器仍然保持视觉主导地位

## 6. 阅读器设计

### 6.1 中间区仅保留阅读器

中间区域改造成单一阅读器，不再承担分析卡片和信息拼盘功能。

新的阅读器中不应再出现：

- `Previous / Current` 并排对比卡
- 摘要模块
- cap table 预览块
- 其他仪表盘式内容块

新的阅读器只保留：

- 极简头部
- 文档页显示区域
- 左右翻页控件

### 6.2 阅读器头部

阅读器头部应保持极简和稳定。

建议内容：

- 当前文档名
- 当前页码 / 总页数
- 上一页按钮
- 下一页按钮

布局要求：

- 控件放在头部，不要悬浮在正文上方
- 不要把交互按钮压在阅读正文之上
- 文档标题存在感要轻，不能抢主阅读区域

### 6.3 阅读表面

正文区域需要具备稳定、舒适的阅读感。

要求：

- 保留充足的内边距
- 文本保持在舒适阅读宽度内
- 主阅读流不再依赖纵向长滚动
- 每次只呈现一页
- 翻页前后边距和容器尺寸保持稳定，避免视觉跳动

### 6.4 分页模型

已确认采用真实分页的左右翻页模式。

行为要求：

- 打开文档后进入分页阅读模式
- 用户通过上一页 / 下一页控件切换页面
- 上下长滚动不再作为默认阅读模式
- 如果文档只有 1 页，左右按钮保留但置灰
- 切换到另一份文档时，阅读器重置到第 1 页

格式处理建议：

- `DOCX` 和 `PDF` 优先按真实分页展示
- 纯文本或派生文本内容需要按页面高度切片，仍然保持“翻页”而不是“滚到底”的交互体验

## 7. 空状态、加载态与错误态

### 7.1 默认空状态

页面初始进入时，不自动打开文档。

阅读器显示已确认的 `A2` 引导型空状态，包含：

- 一条明确的标题，说明阅读器已准备好
- 一条简短说明，提示用户从 `Files` 选择文件
- 可选的次级提示，告诉用户也可以上传新文档

关键约束：

- 这个空状态必须放在完整阅读器壳层内部
- 不能把页面切成另外一种“欢迎页”结构
- 文档打开前后，阅读器尺寸和整体结构要保持稳定

### 7.2 加载态

当文档正在准备时：

- 保持阅读器整体壳层可见
- 在正文区域中央显示加载态
- 页码控件保留，但在内容未准备好前置灰

### 7.3 错误态

当文档无法渲染时：

- 在阅读器正文区域内显示失败状态
- 不跳转到其他模块
- `AI Chat` 和 `Files` 保持原位
- 提供回到文件选择的明确路径

## 8. Files 面板行为

右侧只保留 `Files` 面板。

要求：

- 文件选择仍然是主操作
- 上传入口放在 `Files` 头部区域
- 当前激活文档要高亮显示
- 选中态和未选中态区分清晰
- 面板可折叠，但不再混入无关模块

文件被选中时：

- 在中间阅读器打开该文档
- 默认从第 1 页开始
- 用阅读界面替换空状态

## 9. AI Chat 行为

左侧 `AI Chat` 保留，但视觉层级降为辅助位。

要求：

- 保留现有对话能力
- 保留消息流和输入区
- 通过更暗、更安静的表面降低对阅读器的视觉竞争
- 用户可随时折叠面板换取更大阅读空间

它服务于阅读流程，而不是与阅读器争夺主视觉。

## 10. 动效与视觉层级

整体视觉继续保持当前深色、极简、橙色点缀的语言，但层级要更明确。

指导原则：

- 中间阅读器是最亮、最稳定的表面
- 两侧面板的视觉存在感要弱于阅读器
- 折叠/展开动效要短、轻、克制
- 动效以“帮助定位”为目的，而非装饰
- 继续遵循 reduced motion 约束

## 11. 状态模型调整

重构后不再需要以下动态尺寸状态：

- 左栏宽度
- 右栏宽度
- `History` 高度

新的界面状态重点变为：

- 左侧栏是否折叠
- 右侧栏是否折叠
- 当前选中的文档
- 当前文档所在页码
- 当前文档是否处于空、加载、错误或可阅读状态

## 12. 实现备注

当前 `Dashboard.tsx` 同时混合了：

- 布局
- 上传逻辑
- 预览渲染
- 聊天界面
- topology 浮层
- 对比渲染
- resize 行为

这次改版的实现建议保持聚焦：

- 先简化现有布局逻辑，再接入新的阅读模式
- 删除只服务于拖拽布局和右侧上下堆叠模块的代码
- 不做与本次设计无关的大规模整理，除非它直接服务当前改版

未来可以继续做组件拆分，但不是本次设计的前置条件。

## 13. 风险与边界情况

- 不同文档格式的分页能力可能不一致
- 纯文本类文档可能需要合成分页
- 在较小屏幕下，可能需要在响应式断点上让某一侧默认折叠
- 当前依赖中间区域的 comparison 或 topology 能力，可能需要迁移位置或暂时退出主流程

## 14. 测试策略

建议围绕以下行为做验证：

- `History` 不再渲染
- 界面中不存在 resize handle
- `AI Chat` 与 `Files` 默认均为展开态
- 左右两侧都可以独立折叠和展开
- 折叠按钮在视觉上保持镜像关系
- 选择文件后可以打开阅读器
- 未选中文档时默认显示 `A2` 引导空状态
- 文档导航使用左右分页控件
- 切换文件后页码重置到第 1 页
- 单页文档的翻页控件会正确置灰
- 加载态和错误态都保持在阅读器壳层内部

## 15. 不在本次范围内

本次改版不包括：

- 重做后端文档解析
- 改变聊天语义
- 把 topology 重新设计为主工作区
- 以其他形式恢复可拖拽布局
- 在中间区域重新引入多文档对比工作台
