# 前端待修改清单

本文档只列前端需要跟进的改动，基于本轮后端已修复的 3 个问题整理：

- 上传文件按 basename 落盘导致跨 case / 跨文件夹覆盖
- 删除单个文件夹会清空整个 case 聊天历史
- `set_viewing_version` 未校验 node 是否属于当前 case

另外补充 1 个之前已经确认的前端问题：

- viewing version 后端能力已存在，但前端没有真正消费

## 1. 让 viewing version 真正生效

### 当前问题

前端在 [frontend/src/pages/dashboard/Dashboard.tsx](/Users/mrboring/stanford-law-hackathon/frontend/src/pages/dashboard/Dashboard.tsx) 里只取：

- `snapshot?.captableVersions.find((version) => version.status === "current")`

这意味着界面始终展示 current cap table，而不是当前 workspace 的 `currentViewingNodeId` 对应版本。

### 需要修改

1. 在 Dashboard 中优先根据 `snapshot.workspace.currentViewingNodeId` 或 `snapshot.topology.currentViewingNodeId` 找到对应的 cap table version。
2. 如果当前 viewing node 不是 `captable_version`，要定义清晰兜底策略：
   - 优先展示与该 node 关联的 cap table version；或
   - 退回 current version；或
   - 明确提示当前节点没有 cap table 视图。
3. 不要再单纯以 `status === "current"` 作为主选择条件。

### 影响文件

- [frontend/src/pages/dashboard/Dashboard.tsx](/Users/mrboring/stanford-law-hackathon/frontend/src/pages/dashboard/Dashboard.tsx)
- 如需补 selector，也可能涉及 [frontend/src/context/WorkbenchContext.tsx](/Users/mrboring/stanford-law-hackathon/frontend/src/context/WorkbenchContext.tsx)

## 2. 前端补出 viewing version 切换入口

### 当前问题

虽然 `WorkbenchContext` 已经暴露了 `setViewingVersion`，但 Dashboard 当前没有实际入口让用户切换历史版本。

### 需要修改

1. 在右侧文件/版本面板或新增的 topology/history 面板里，给 topology node 或 cap table version 加上“View”操作。
2. 点击后调用 `setViewingVersion(caseId, nodeId)`。
3. 切换成功后，以下区域需要同步刷新语义：
   - 当前 cap table
   - 当前文档/版本标题
   - 当前 viewing version 标识
   - 操作提示或状态标签

### 建议

- 当前 UI 里已经有 document / cap table 双视图，比较适合新增一个轻量的 “Version / History” 区块，而不是继续把逻辑埋在隐式状态里。

## 3. 区分 current version 和 viewing version

### 当前问题

即使后端记录了：

- `currentTrunkNodeId`
- `currentViewingNodeId`

前端界面没有把两者区别展示出来。用户无法知道自己看到的是：

- 当前 trunk
- 某个历史 viewing version
- 某个 draft merge 后的版本

### 需要修改

1. 在工作区 header 或 cap table header 增加状态说明：
   - `Current trunk`
   - `Viewing historical version`
   - `Viewing derived version`
2. 当 viewing node 不等于 trunk node 时，明显提示用户当前正在看历史视图。
3. 提供“回到当前版本”按钮。

### 影响文件

- [frontend/src/pages/dashboard/Dashboard.tsx](/Users/mrboring/stanford-law-hackathon/frontend/src/pages/dashboard/Dashboard.tsx)

## 4. 删除文件夹后，不要误导用户以为聊天被清空

### 当前问题

后端已经修复 folder 删除不再清空 `case_messages`，但前端文案仍然写着删除文件夹会删除整个 case chat context。

2. 明确区分：
   - 删除文件夹：删除该文件夹内文档及衍生产物，不删除整个 workspace 聊天历史
   - 删除 workspace：删除整个 case 的文件、结果、拓扑和聊天历史

### 具体位置

## 5. 上传成功后的前端状态，不要依赖后端物理文件名

### 当前问题

这轮后端修复后，物理存储文件名不再等于用户上传文件名。前端如果在任何地方假设：

- 返回的 `source_file`
- 下载文件名
- 本地展示文件名

三者是同一个字段，就容易出现展示或定位混乱。

### 需要确认和调整

1. 列表展示继续使用 `document.fileName`，不要展示内部落盘名。
2. 上传反馈区如果展示 `processed[].source_file`，应确认它显示的是原始文件名，而不是物理文件名。
3. 若后续增加“重新上传同名文件”或“按文件名聚合”的逻辑，必须用：
   - `document.id`
   - `relativePath`
   - `identityKey`
   而不是只用 `fileName`。

## 6. 错误提示需要覆盖新的后端校验语义

### 当前问题

后端现在会拒绝跨 case 的 `set_viewing_version`。前端如果以后加入 viewing version 切换入口，需要能正确处理这种失败，而不是只显示笼统报错。

### 需要修改

1. 对 `setViewingVersion` 失败增加更明确提示。
2. 如果后端返回 404 / validation 类错误，要提示：
   - 当前版本节点不存在；或
   - 当前节点不属于此 workspace；或
   - 请刷新工作区后重试

## 7. 建议顺序

建议按下面顺序改前端：

1. 修正 folder 删除弹窗文案
2. 让 cap table 读取 `currentViewingNodeId`
3. 增加 viewing version 切换入口
4. 增加“当前 trunk / 当前 viewing”状态提示
5. 补错误提示和回退逻辑

## 8. 最低验收标准

前端改完后，至少应满足：

1. 删除单个文件夹后，聊天记录仍保留，且文案不误导。
2. 用户可以切换到历史 viewing version。
3. 切换后 cap table 内容确实变化，而不是始终显示 current version。
4. 用户能看懂自己当前是在看 trunk 还是历史版本。
5. 同名文件上传后，前端展示和交互不受后端内部落盘文件名影响。
