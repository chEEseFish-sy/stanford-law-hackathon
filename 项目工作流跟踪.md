## 3. 本轮优化摘要

- 更新 1：通读拓扑技术说明并将其前端化
  - 以 `TopologyNode`、`DocumentMeta`、`StructuredResult`、`CapTableVersion` 为统一前端类型契约
  - 将主干、草案、已合并、被否决、已归档、处理中等状态映射到图节点与详情 Portal
- 更新 2：重构前端工作台为统一 workbench
  - Dashboard、Document Intake、Evidence Review、Event Timeline、Working Cap Table、Topology Graph 现在共享同一份 workspace snapshot
  - 当前 viewing version 与 current trunk 已联动显示
- 更新 3：预留后端接口
  - 已在前端 API 层补齐获取拓扑、节点详情、文件上传、合并、否决、归档、切换 viewing version 等接口契约
  - 当前使用 mock service 驱动，后续可直接替换为真实后端
- 更新 4：增强回溯与详情能力
  - 单击节点高亮相关路径
  - 双击节点打开右侧详情 Portal
  - Portal 可展示文件概况、结构化结果、Cap Table 版本、相关节点和可执行操作

## 4. 历史迭代记录

### Step 1：确定产品方向

- 从多个方向收敛到股权审计与 cap table reconciliation
- 明确系统围绕交易文件、证据审理、股权事件重建和 cap table 对账展开

### Step 2：明确用户与场景

- 识别 primary persona 为融资律师 / paralegal
- 识别 secondary persona 为 small startup founder / finance ops / in-house legal
- 明确主要场景为融资前核查、尽调前预审、文档版本变化审理

### Step 3：完成市场调研

- 输出市场调研报告
- 确认产品位于 LegalTech、尽调工作流、股权管理交叉带
- 确认市场空白在“法律证据驱动的股权重建 + 异常审计”

### Step 4：完成初版产品设计

- 输出初版产品设计文档
- 明确核心工作流：
  - 文件整理
  - 证据审理
  - 股权事件标准化
  - working cap table 生成
  - cap table 对账
  - 文档版本对比
  - 版本回溯

### Step 5：识别文档问题

- 发现初版文档主要问题为：
  - 结构不够像正式产品方案
  - 范围边界和优先级不够强
  - 与市场调研的差异化衔接不够明显
  - 给 UX / 开发的 handoff 不够直接

### Step 6：完成本轮统一命名与结构升级

- 将产品统一命名为 VeriCap
- 重写产品设计方案
- 优化项目工作流跟踪文档
- 重新生成正式 docx 交付物

### Step 7：接入拓扑技术设计到前端

- 读取拓扑技术说明文档与现有 docx 设计资料
- 在前端建立共享 workbench 数据模型与 mock workspace
- 重构拓扑图，使其表达主干、分支、引用关系、冲突关系和历史回溯
- 为后续后端接口补齐契约层与状态流转入口
- 完成构建验证并保持可预览

  - 权限控制
  - 审批流
  - 审计日志

## 8. 当前边界提醒

- VeriCap 强调 audit copilot，不强调 legal decision maker
- 当前输出是 working cap table，不是最终法律真相
- 当前版本优先做高置信度、强解释性、可回溯的核心闭环
- 当前不追求复杂证券全覆盖与企业级系统集成
- 拓扑模块只负责结构关系与业务入口，不直接保存完整文件内容或完整结构化 JSON

## 9. 后续执行约束

- 任何后续设计都必须明确触发时机、返回路径与状态反馈，不能只写功能清单
- 后续 UX 设计必须围绕“风险优先级、冲突处理、证据追溯、版本变化”展开，不能退化为静态表格展示
- 开发实现必须优先搭建文件整理、证据状态、事件模型、working cap table、拓扑图、版本回溯与对账逻辑
- 每次文档更新后都要重新生成相关 docx，并刷新本文件中的修改时间与状态
